# 多 Agent Multitask 优化小程序 — 总控 Prompt

## 用途

可直接粘贴到 Cursor（开启 **Multitask**）的总控 Prompt，用于对微信小程序 **dos-quiz** 做多维优化：并行专家调研 → 汇总变更清单 → 三主审门禁 → 实施 → 发布说明。

目标读者：需要复用「发布门禁制」协作流程的开发者 / Agent。

## workshop 约定

| 路径 | 性质 |
|------|------|
| `workshop/` | **开发空间**：中间探索、策展、prompt、草稿、原文题库 |
| `miniprogram/` | **主包代码**（微信打包入口，见 `project.config.json` 的 `miniprogramRoot`） |

- `workshop/` **不进主包**（不会被小程序打包），可放心放 prompt、题库原文、策展草稿。
- 本文件路径（相对仓库根）：`workshop/docs/multitask-optimize-agents.md`
- 原文题库参考：`workshop/question/questions.txt`
- 主包题库：`miniprogram/utils/questions-data.js`

## 如何在 Cursor Multitask 使用

1. Cursor 开启 **Multitask**，把下方「总控 Prompt」整段贴进对话。
2. 若只要「出方案不改码」，保留文末「确认前不要写代码」。若要自动改到发布，改成：「三主审全票后无需再问，直接 Phase D–E」。
3. 范围太大时，在开头加一句限定，例如：`本轮只做 P0 + 题库/答题页，不动社区页。`
4. 路径均相对本仓库根目录 `dos-quiz/`。

# 总控 Prompt（可复制）

~~~~markdown
# 任务：多 Agent 协作优化微信小程序 dos-quiz（发布门禁制）

## 项目
- 仓库：dos-quiz（微信小程序）
- 主路径：`miniprogram/`（重点 `pages/quiz`、`pages/profile`、`pages/community`、`utils/`）
- 云函数：`cloudfunctions/`
- 题库：`miniprogram/utils/questions-data.js`（原文题库参考 `workshop/question/questions.txt`）
- 开发空间：`workshop/`（prompt / 策展 / 草稿，不进主包）
- 约束：主包 ≤2MB；云图须经云函数换临时链，禁止把 `cloud://` 直接塞给 `<image>`；**未三方一致通过前禁止 commit/push/发布**

## 总目标（多维优化）
从以下维度提出可落地改动并最终实施（仅在门禁通过后）：
1. **趣味性**：沉浸、钩子、节奏、分享欲
2. **可玩性**：流程清晰、反馈及时、收集/结果有意义、少卡顿
3. **严谨性**：文案紧贴陀氏原文思想、计分/轴映射自洽、字段完整
4. **安全性**：隐私、存储、云权限、注入/越权、敏感内容边界
5. **系统架构**：模块边界、包体积、云调用、缓存、可维护性

## 工作模式（必须遵守）
1. **先并行、后汇合**：各专家角色**独立**完成各自任务，互不代写对方结论；可用并行 subagent。
2. **只读调研 → 产出报告 → 汇总方案 → 三主审 → 实施 → 再三主审 → 发布**。
3. 改代码前必须有「变更清单」；禁止无关重构；禁止擅自改 git config / force push。
4. **文案原则**：
   - 题干（`scene` / `quote`）尽量用 `workshop/question/questions.txt` **原文**；过长只做必要删减，不改写语气、不换成白话隐喻解释。
   - 选项去掉解释性句子，只留干净立场（删「因为/证明/哪怕/至少」等后半句）。
5. 用中文写报告；每个角色输出必须结构化（问题 / 证据路径 / 建议 / 优先级 P0–P2 / 验收标准）。

---

## Phase A：专家角色并行（各自独立，只读 + 出方案，不改主分支）

请启动以下 **独立 Agent**，每人一份 brief，互不合并结论：

### A1 系统架构师
- 审：`miniprogram/app.js`、页面路由/tabBar、`miniprogram/utils/` 分层、云函数调用链、包体积与资源加载
- 产出：架构现状图（文字）、瓶颈、模块边界建议、P0 架构债清单

### A2 艺术评审
- 审：视觉层次、答题/结果/我的页氛围、动效是否噪音、文案可读性与调性
- 产出：体验问题清单 + 不破坏品牌调性的改法（避免违和背景图、避免堆特效）

### A3 项目策划
- 审：信息架构（测试/社区/我的）、用户路径、收集与掉落是否有意义、社区占位策略
- 产出：一页产品方案（目标用户、核心回路、本迭代只做/不做）

### A4 测试工程师
- 审：关键路径（开始→答题→结果→分享→图鉴）、边界（返回、重开、无网、云失败）
- 产出：测试用例表 + 已知风险点 + 手工回归清单

### A5 安全工程师
- 审：本地存储、云环境 ID、云函数鉴权、临时 URL、用户数据、敏感题内容
- 产出：威胁模型简表 + 必须修的安全项（P0）

### A6 算法工程师
- 审：题目选项 → axes / characterId → 主人格/影子 → 雷达/相似度（`miniprogram/utils/` 计分相关）
- 产出：计分正确性检查、偏差/同质化风险、校验脚本或自检步骤建议

### A7 心理学家（娱乐向人格测合规）
- 审：结果文案是否过度诊断、免责声明、用户情绪风险、措辞边界
- 产出：文案/流程合规建议（保持娱乐向，禁止医疗暗示）

（可选加派）**文案/文学校对**：对照 `workshop/question/questions.txt`，标出偏离原文或选项解释腔残留。

每个角色交付格式：
```
## 角色
## 范围与方法
## 发现（带文件路径）
## 建议改动（P0/P1/P2）
## 明确不改项
## 验收标准
```

---

## Phase B：汇总（1 个整合 Agent）
- 合并 A1–A7，去重冲突，输出 **《本迭代变更清单》**：
  - 必做 P0（安全/崩溃/计分错误）
  - 应做 P1（趣味/可玩/架构小改）
  - 不做清单（防范围膨胀）
- 给出预计改动文件列表与回归点
- **至此仍不改代码、不发布**

---

## Phase C：三主审门禁（3 个主 Agent，必须独立评审）

启动三个**互不共享草稿结论**的主审 Agent，各自阅读《变更清单》+ 关键代码现状，只投票：

| 主审 | 关注点 | 通过条件 |
|------|--------|----------|
| **主审1 · 产品体验主审** | 趣味性、可玩性、艺术调性、策划目标是否对齐 | 无「更难懂/更无聊/违和」的 P0 |
| **主审2 · 工程可靠主审** | 架构、性能、包体积、测试覆盖、可回滚 | 无未测高风险改动；有明确回归清单 |
| **主审3 · 安全合规主审** | 安全、隐私、敏感内容、心理学边界、文案严谨性 | 无未处理安全 P0；无过度诊断措辞 |

每主审必须输出：
```
裁决：APPROVE | REQUEST_CHANGES | REJECT
理由（≤10 条）
必须修改项（若有）
对发布的明确意见：允许 / 禁止
```

### 门禁规则（硬性）
- **仅当 3 个主审全部为 APPROVE** → 进入 Phase D 实施与发布
- 任一 `REQUEST_CHANGES` → 打回修改，改完后 **重新走三主审**（不可只找原反对者补票）
- 任一 `REJECT` → 停止本迭代发布，只保留安全热修（需三主审再批热修范围）
- 禁止用「平均分」「多数通过」代替全票通过

---

## Phase D：实施（仅门禁通过后）
1. 按《变更清单》P0→P1 改代码；最小 diff；匹配现有风格
2. 跑通自检：题库 20 题字段、关键路径、云 URL 解析失败兜底
3. **实施后再请同一套三主审做「发布前复审」**（看 diff，不是看方案）
4. 复审再次 **3×APPROVE** 后才允许：
   - `git commit`（中文清晰 message）
   - `git push origin main`（或用户指定分支）
   - 提醒用户：微信开发者工具重新编译；云函数若有改动需上传部署

若复审未过：撤回或追加修复，不得发布。

---

## Phase E：发布说明（给用户）
- 改了什么（bullet）
- 三主审裁决摘要（各一句）
- commit hash
- 用户侧验证清单（5 条以内）

2. 等齐后做 Phase B 汇总
3. 启动 Phase C 三主审
4. **在我确认三主审结果前，不要写代码、不要 commit**（除非我另行说「按通过项直接做」）
~~~~
